# Skill Writing Guide

这份指南用于维护 `superskills` 中的每一个 skill，确保它们不是零散 prompt，而是稳定、可复用、可迭代的工作流。

## 什么是一个好 skill

一个好 skill 应该满足：

1. **目标清晰**：明确解决什么问题。
2. **输入明确**：知道用户需要提供哪些信息。
3. **输出稳定**：每次回答结构一致，方便复制和执行。
4. **约束具体**：告诉 AI 什么不能做，避免废话和跑题。
5. **示例充足**：通过 examples 固定质量标准。
6. **可迭代**：记录版本变化，持续优化。

## 推荐文件结构

```txt
skill-name/
├── skill.md
├── prompt-template.md
├── examples.md
└── changelog.md
```

## skill.md 应该包含什么

- Skill 名称
- 角色定义
- 使用目标
- 适用场景
- 输入字段
- 输出格式
- 默认假设
- 硬性约束
- 质量标准
- System Prompt

## prompt-template.md 应该包含什么

- 一份最常用的任务输入模板
- 一份极简输入模板
- 一份高级输入模板

## examples.md 应该包含什么

- 高质量输入示例
- 高质量输出示例
- 不推荐的差输出
- 常见场景变体

## changelog.md 应该包含什么

- 每次更新的日期
- 改了什么
- 为什么改
- 对输出质量有什么影响

## 编写原则

### 1. 用约束代替情绪化描述

不好：

```txt
回答要高级一点，不要太普通。
```

好：

```txt
回答必须包含 3 个可执行方案，每个方案包含名称、适用场景、步骤、风险和下一步行动。
```

### 2. 固定输出结构

不好：

```txt
帮我想一些创意。
```

好：

```txt
每个创意必须包含：名称、目标用户、视觉元素、执行步骤、可选变体。
```

### 3. 明确反例

告诉 AI 不要做什么通常比只说“做好一点”更有效。

例如：

```txt
不要输出泛泛的建议，不要解释基础概念，不要给我无法执行的灵感词。
```

### 4. 保留默认假设

如果用户没有提供完整信息，skill 应该有默认行为。

例如：

```txt
如果用户没有说明尺寸，默认按 100 × 100 mm 设计。
```

## 推荐版本节奏

- `v0.1`：能用，结构清楚。
- `v0.2`：补充真实 examples。
- `v0.3`：加入反例和质量检查清单。
- `v1.0`：稳定，可长期复用。
